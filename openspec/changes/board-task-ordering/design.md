## Context

Текущая реализация досок (Date Board и Status Board) поддерживает перетягивание задач только между колонками для изменения `planned_date` или `status_id`. Порядок задач внутри колонки определяется массивом из API и опциями сортировки (дедлайн, приоритет), но ручной порядок не сохраняется.

Обе доски используют нативный HTML5 Drag and Drop API (не `DragDropManager` из `utils/drag-drop.ts`). Задачи оборачиваются в `.draggable-task` div, drop-зоны — `.column-content[data-drop-zone]`.

В модели `Task` нет поля для хранения пользовательского порядка. Спецификация `task-views` уже содержит сценарий "Manual order preserved", но он не реализован.

## Goals / Non-Goals

**Goals:**
- Пользователь может перетягивать задачи внутри одной колонки для изменения их порядка
- Пользователь может перетягивать задачи между колонками с размещением в конкретной позиции
- Порядок задач сохраняется на сервере и восстанавливается при перезагрузке
- Ручной порядок работает когда выбрана сортировка "По умолчанию"
- Автоматические сортировки (дедлайн, приоритет) имеют приоритет над ручным порядком

**Non-Goals:**
- Изменение порядка задач в виде "Списки" (отдельная задача)
- Анимация перетягивания (drag placeholder/ghost) — минимальная визуальная обратная связь
- Офлайн-очередь для reorder-операций
- Группировка по подзадачам при reorder

## Decisions

### D1: Поле `sort_order` в модели Task

**Решение**: Добавить `sort_order` (Integer, default=0, nullable=False) в модель `Task`.

**Альтернативы**:
- Отдельная таблица `task_board_order` — сложнее, требует JOIN, нет явной выгоды для KISS
- JSON-массив порядка в настройках — нет транзакционности, сложно при параллельных изменениях

**Обоснование**: Простое целочисленное поле в `Task` — самый простой подход. Значение вычисляется как позиция в колонке. При переносе между колонками обновляются только затронутые задачи.

### D2: Эндпоинт `PATCH /api/tasks/reorder`

**Решение**: Создать пакетный эндпоинт для обновления порядка нескольких задач за один запрос.

Формат запроса:
```json
{
  "items": [
    {"task_id": 1, "sort_order": 0},
    {"task_id": 2, "sort_order": 1},
    {"task_id": 3, "sort_order": 2}
  ]
}
```

**Альтернативы**:
- Индивидуальные `PATCH /api/tasks/{id}` для каждой задачи — N запросов при перестановке
- Отправка полного порядка колонки — избыточно

**Обоснование**: Пакетный эндпоинт минимизирует количество запросов и гарантирует атомарность обновления порядка. Клиент отправляет только изменившиеся позиции.

### D3: Определение позиции вставки на фронте

**Решение**: При `dragover` над задачей в колонке определять верхнюю/нижнюю половину элемента для вставки до/после. Показывать CSS-линию-индикатор (placeholder) в точке вставки.

**Обоснование**: Стандартный паттерн для drag-and-drop списков. Не требует внешних библиотек.

### D4: Стратегия обновления порядка

**Решение**: При drop-событии клиент вычисляет новый порядок задач в колонке и отправляет массив `{task_id, sort_order}` на `/api/tasks/reorder`. Используются шаги 1000 (gap-based) для минимизации обновлений — но для простоты используем последовательные числа (0, 1, 2...) и обновляем все задачи в колонке после точки вставки.

**Обоснование**: Проще в реализации и отладке. Для типичного количества задач в колонке (< 50) производительность не является проблемой.

## Risks / Trade-offs

- **[Конкуренция при одновременном редактировании]** → Порядок привязан к пользователю (`user_id`), риск минимален. При конфликте последний write выигрывает.
- **[Производительность при большом количестве задач]** → Обновление порядка затрагивает только одну колонку. Для < 100 задач в колонке — не проблема.
- **[Миграция: существующие задачи без sort_order]** → Миграция устанавливает `sort_order = 0` для всех существующих задач. После миграции порядок определяется `created_at` как fallback (задачи с одинаковым sort_order сортируются по дате создания).
