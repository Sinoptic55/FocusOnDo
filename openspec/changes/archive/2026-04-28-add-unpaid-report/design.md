## Context

Система Pomodoro TMS уже имеет модуль аналитики (`src/backend/api/analytics.py`) с отчётами по проектам, клиентам, продуктивности и другим метрикам. Фронтенд использует компонент `analytics-reports.ts` для отображения отчётов с вкладками.

В моделях задач (`Task`) есть флаги `is_completed` и `is_paid`. Сегменты времени (`TimeSegment`) хранят `actual_time_seconds` и `billed_time_seconds`. Требуется добавить отчёт по выполненным, но не оплаченным задачам.

## Goals / Non-Goals

**Goals:**
- Создать backend endpoint `GET /api/analytics/unpaid-tasks` с фильтрами по project_id, client_id, list_id
- Создать backend endpoint `GET /api/analytics/unpaid-tasks/export` для генерации ODS-файла
- Создать frontend компонент `unpaid-tasks-report.ts` с таблицей и фильтрами
- Интегрировать новый отчёт в существующую страницу аналитики

**Non-Goals:**
- Изменение модели данных — используются существующие таблицы
- Автоматическое выставление счетов — только отчёт
- Поддержка других форматов экспорта (CSV, XLSX) — только ODS

## Decisions

### 1. Backend: отдельный endpoint для данных и экспорта

**Решение:** Два endpoint'а — `/api/analytics/unpaid-tasks` (JSON) и `/api/analytics/unpaid-tasks/export` (ODS blob).

**Альтернатива:** Один endpoint с параметром `?format=ods`. Отклонено — разные content-type и логика обработки усложнят код.

### 2. Фильтрация на уровне SQL

**Решение:** Фильтры project_id, client_id, list_id применяются на уровне SQL-запроса через `WHERE` условия. Это эффективно для больших объёмов данных.

### 3. ODS генерация через odfpy

**Решение:** Использовать библиотеку `odfpy` — стандарт для работы с ODF в Python. Она поддерживает создание таблиц, стилизацию и корректную кодировку UTF-8.

**Альтернатива:** `pyexcel-ods3` — менее гибкая, `openpyxl` — только XLSX.

### 4. Фронтенд: отдельный компонент с интеграцией в analytics-reports

**Решение:** Создать `unpaid-tasks-report.ts` как самостоятельный компонент и добавить вкладку "Неоплаченные" в `analytics-reports.ts`.

### 5. Критерий "не оплачено"

**Решение:** Задача считается не оплаченной, если `is_paid = false` И `is_completed = true`. Это соответствует существующей модели данных.

## Risks / Trade-offs

| Risk | Mitigation |
|------|-----------|
| Большой объём данных при экспорте | Ограничить экспорт последними 1000 записями; добавить предупреждение |
| odfpy — дополнительная зависимость | Библиотека стабильная, с поддержкой Python 3.10+ |
| Длительные SQL-запросы при большом количестве сегментов | Добавить индексы на `tasks.is_completed`, `tasks.is_paid`, `time_segments.task_id` |
| Кодировка кириллицы в ODS | odfpy корректно обрабатывает UTF-8; протестировать при реализации |

## Migration Plan

1. Добавить `odfpy` в `requirements.txt`
2. Создать новый endpoint в `analytics.py`
3. Создать frontend компонент
4. Добавить вкладку в analytics-reports
5. **Rollback:** Удалить endpoint, компонент, зависимость — без влияния на существующий функционал

## Open Questions

- Нужно ли учитывать подзадачи (steps) отдельно или агрегировать время на уровне родительской задачи? **Решение:** Агрегировать на уровне родительской задачи (parent_task_id IS NULL).
- Нужна ли пагинация в JSON-ответе? **Решение:** Нет, отчёт обычно содержит десятки записей; пагинация не требуется.
