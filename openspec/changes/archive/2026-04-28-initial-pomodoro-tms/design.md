## Context

Создаётся с нуля веб-система управления задачами с интегрированным Помодоро таймером. Текущая кодовая база пуста. Проект следует принципу KISS (Keep It Simple, Stupid) — предпочтение простым решениям. Целевая платформа: Windows Server с браузерным доступом, в будущем — desktop и mobile приложения.

Основные характеристики:
- Однопользовательская система на первом этапе
- Серверное хранение данных с интернет-доступом
- Real-time обновление таймера и времени на задачах
- Три режима визуализации задач с разной логикой группировки

## Goals / Non-Goals

**Goals:**
- Создать работающий MVP всех 10 capabilities из proposal
- Простая, понятная архитектура без переусложнения
- Чистое разделение frontend/backend через REST API
- Нормализованная схема БД с поддержкой всех фич
- Персистентность таймера (продолжает работать при навигации по UI)
- Real-time обновление данных (время работы таймера, суммарное время)

**Non-Goals:**
- Многопользовательская совместная работа и права доступа (только изоляция данных между пользователями)
- Offline-режим и синхронизация
- Масштабирование на тысячи пользователей (оптимизация на малую/среднюю нагрузку)
- Мобильные и desktop приложения (только веб на первом этапе)
- WebSocket/SSE для push-уведомлений (polling на клиенте для real-time)
- Интеграции с внешними системами (только ссылка на внешнюю задачу)

## Decisions

### 1. Frontend: Vanilla TypeScript + современный подход

**Решение**: TypeScript без крупных фреймворков (React/Vue/Angular). Модульная структура с компонентным подходом.

**Обоснование**:
- Принцип KISS — нет лишних зависимостей
- TypeScript даёт типобезопасность и лучший DX
- Нативные Web Components или простые TS классы компонентов
- Меньше сложности сборки, быстрее старт

**Альтернативы**:
- React — избыточность для однопользовательского приложения, усложняет простоту
- Vue — хорош но добавляет зависимость, которая не критична на данном этапе
- Svelte — компактнее фреймворков, но менее распространён

**Структура frontend**:
```
src/frontend/
  ├── components/     # UI компоненты (таймер, задачи, доски)
  ├── services/       # API клиенты, бизнес логика
  ├── models/         # TypeScript типы и интерфейсы
  ├── views/          # Страницы (списки, доски, настройки)
  ├── utils/          # Хелперы (форматирование времени, цвета)
  └── main.ts         # Entry point
```

### 2. Backend: FastAPI (Python)

**Решение**: FastAPI как веб-фреймворк.

**Обоснование**:
- Современный, быстрый, с автоматической документацией API
- Нативная поддержка async/await для производительности
- Автоматическая валидация через Pydantic
- Встроенная интеграция с PostgreSQL через asyncpg
- Простая авторизация через middleware

**Альтернативы**:
- Flask — проще но медленнее, нет встроенной валидации и async
- Django — избыточен для API-only backend, много магии

**Структура backend**:
```
src/backend/
  ├── api/            # API endpoints (роутеры по доменам)
  ├── models/         # SQLAlchemy ORM модели
  ├── schemas/        # Pydantic схемы для валидации
  ├── services/       # Бизнес логика (вне эндпоинтов)
  ├── db/             # Database connection, migrations
  ├── auth/           # Аутентификация и middleware
  └── main.py         # FastAPI app
```

### 3. База данных: PostgreSQL с нормализованной схемой

**Решение**: PostgreSQL 12+ с полной нормализацией данных.

**Ключевые таблицы**:
- `users` — пользователи (id, username, password_hash, settings_json)
- `tasks` — задачи (id, user_id, title, description, planned_date, deadline, status_id, list_id, project_id, client_id, pomodoro_estimate, first_action, external_link, parent_task_id для иерархии)
- `task_lists` — списки задач (id, user_id, name, color)
- `task_statuses` — статусы (id, user_id, name, board_visible, order)
- `projects` — проекты (id, user_id, name, color)
- `clients` — клиенты (id, user_id, name)
- `time_segments` — отрезки работы (id, task_id, user_id, start_time, duration_seconds, actual_time_seconds, billed_time_seconds, energy_level, task_progressed, stuck)
- `pomodoro_intervals` — настройка интервалов (id, user_id, order, type[work/break], duration_minutes)
- `recurring_tasks` — настройки повторений (id, task_id, frequency_type, frequency_data_json, last_created_date)
- `app_settings` — настройки (id, user_id, settings_json: тема, звуки, горячие клавиши и т.д.)

**Индексы**:
- tasks: (user_id, planned_date), (user_id, status_id), (user_id, list_id)
- time_segments: (task_id), (user_id, start_time)

**Обоснование**:
- PostgreSQL надёжен, поддерживает JSONB для гибких настроек
- Полная нормализация для целостности данных
- parent_task_id в tasks для иерархии (шаги = задачи с parent_task_id)

**Альтернативы**:
- SQLite — не подходит для серверного развёртывания с интернет-доступом
- MongoDB — избыточно для структурированных данных, слабее в транзакциях

### 4. Архитектура таймера: Client-side с backend персистенцией

**Решение**: Таймер работает в браузере (JavaScript setInterval), состояние сохраняется в localStorage + синхронизируется с сервером.

**Поток**:
1. Таймер запускается — создаётся сессия в localStorage (active_task_id, interval_type, start_time, duration)
2. JS setInterval каждую секунду обновляет UI
3. При завершении интервала — POST /api/time-segments создаёт запись о времени
4. При паузе/смене задачи — также фиксируется время + обновление localStorage
5. При перезагрузке страницы — восстановление из localStorage

**Обоснование**:
- Не требует WebSocket — проще реализация
- Персистентность через localStorage + periodic sync с сервером
- Минимальная нагрузка на сервер (запись только по завершению интервала)

**Trade-off**: Если пользователь откроет систему в двух вкладках — возможен конфликт. Решение: показывать предупреждение "таймер уже работает в другой вкладке" при обнаружении активной сессии.

### 5. Real-time обновление: Client-side polling

**Решение**: Компонент задачи с активным таймером обновляет своё отображение времени каждую секунду локально (не ходит на сервер).

**Обоснование**:
- Простота реализации
- Нет необходимости в WebSocket/SSE на начальном этапе
- Для одного пользователя достаточно клиентской реактивности

**Альтернативы**:
- WebSocket — избыточно для однопользовательской системы
- SSE — проще WebSocket, но всё равно добавляет сложности

### 6. Аутентификация: Session-based с JWT

**Решение**: JWT токены в HTTP-only cookies для аутентификации.

**Поток**:
1. POST /api/auth/login → получение JWT токена → установка в HTTP-only cookie
2. Все защищённые эндпоинты проверяют токен через middleware
3. Токен содержит user_id, expiration
4. POST /api/auth/logout → очистка cookie

**Обоснование**:
- HTTP-only cookie защищает от XSS атак
- JWT не требует серверного хранения сессий
- Простота реализации в FastAPI через dependency injection

**Альтернативы**:
- Session-based с Redis — избыточно, требует дополнительную зависимость
- OAuth2 — избыточно для простой системы логин/пароль

### 7. Умные инструменты: Реализация на backend

**Решение**:
- "Не знаю с чего начать" → GET /api/smart/next-task → алгоритм на сервере (сортировка: просроченные → сегодня с высоким приоритетом → остальные)
- Быстрый захват → mini-окно (popup) → POST /api/tasks/quick с минимальными полями → сохранение в список "Inbox"
- Утренний ритуал → проверка последнего запуска в app_settings, показ экрана планирования при первом заходе за день
- Недельный обзор → аналогично, напоминание в настроенный день недели
- Детекция застревания → подсчёт "stuck" в time_segments для задачи, уведомление при превышении порога

**Обоснование**:
- Бизнес-логика на сервере легче тестировать и поддерживать
- Frontend только получает результат и показывает UI

### 8. Три вида визуализации: Клиентская фильтрация/группировка

**Решение**: Сервер отдаёт все задачи пользователя (или с фильтром), клиент группирует по логике вида.

**Списки**: группировка по `list_id`, сортировка внутри списка по order/date
**Доска по дате**: группировка по колонкам (нет даты / сегодня / завтра / неделя / позже / когда-нибудь) на основе `planned_date`
**Доска по статусу**: группировка по `status_id`, только статусы с `board_visible=true`

**API эндпоинты**:
- GET /api/tasks?view=list — все задачи пользователя
- GET /api/tasks?view=date-board — те же данные, клиент группирует
- GET /api/tasks?view=status-board — фильтр по board_visible статусам

**Обоснование**:
- Меньше дублирования логики
- Клиент гибко перестраивает UI без запросов к серверу

**Альтернативы**:
- Серверная группировка — избыточные запросы при переключении видов

### 9. Аналитика: Агрегация на SQL-запросах

**Решение**: Эндпоинты аналитики выполняют агрегированные запросы к `time_segments` с JOIN к tasks/projects/clients.

**Примеры**:
- Время по проектам: `SELECT project_id, SUM(actual_time_seconds), SUM(billed_time_seconds) FROM time_segments JOIN tasks GROUP BY project_id`
- Пики продуктивности: группировка по часу дня + фильтр по `energy_level='high'`
- Точность оценки: сравнение `pomodoro_estimate` с фактическим временем из time_segments

**Обоснование**:
- PostgreSQL эффективно справляется с агрегацией на малых/средних объёмах данных
- Нет необходимости в отдельном хранилище для аналитики на начальном этапе

**Trade-off**: При росте данных (годы использования) запросы могут замедлиться. Решение: партиционирование таблицы time_segments по дате, индексы на start_time.

### 10. Миграции БД: Alembic

**Решение**: Alembic для управления миграциями схемы.

**Обоснование**:
- Стандарт для SQLAlchemy
- Версионирование схемы, возможность отката
- Автогенерация миграций из моделей

## Risks / Trade-offs

**[R1] Конкурентный доступ из нескольких вкладок**  
→ **Mitigation**: Детекция активного таймера через localStorage broadcast channel, показ предупреждения пользователю

**[R2] Потеря состояния таймера при закрытии браузера**  
→ **Mitigation**: Periodic sync localStorage → сервер каждые 30 секунд, восстановление при reopening

**[R3] Производительность аналитики при большом объёме данных (годы использования)**  
→ **Mitigation**: Индексы на time_segments(start_time), возможное партиционирование таблицы в будущем

**[R4] Vanilla TypeScript без фреймворка — больше boilerplate кода**  
→ **Trade-off**: Принято сознательно в угоду простоте и отсутствию зависимостей. Если проект вырастет — возможна миграция.

**[R5] Client-side polling вместо WebSocket — задержка обновления данных между вкладками**  
→ **Trade-off**: Принято для упрощения. На начальном этапе пользователь работает в одной вкладке.

**[R6] JWT в cookies — невозможность инвалидации токена до истечения**  
→ **Mitigation**: Короткий TTL токена (24 часа), refresh token механизм для продления сессии

**[R7] Отсутствие offline-режима**  
→ **Trade-off**: Сознательное ограничение требований. Веб-приложение требует интернет.

## Migration Plan

**Initial deployment** (не миграция, а первичное развёртывание):

1. **Database setup**:
   - Установка PostgreSQL 12+
   - Создание БД и пользователя
   - Выполнение начальных миграций Alembic (создание схемы)

2. **Backend deployment**:
   - Установка Python 3.9+, зависимостей (requirements.txt)
   - Настройка переменных окружения (DB connection, JWT secret)
   - Запуск FastAPI через Uvicorn
   - Reverse proxy (nginx) перед FastAPI

3. **Frontend deployment**:
   - Сборка TypeScript → JavaScript bundle
   - Размещение статики (HTML/CSS/JS) на nginx или в S3/CDN

4. **Testing**:
   - Smoke tests: создание пользователя, задачи, запуск таймера
   - Проверка всех трёх видов визуализации

5. **Rollback strategy**:
   - На начальном этапе (нет пользователей) — полная перестановка
   - В будущем — версионирование releases, Alembic downgrade для БД

## Open Questions

**Q1**: Использовать ли TypeScript strict mode?  
→ **Предложение**: Да, для максимальной типобезопасности

**Q2**: Нужен ли rate limiting на API эндпоинты?  
→ **Предложение**: На начальном этапе нет (однопользовательская система), добавить перед открытием многим пользователям

**Q3**: Формат хранения настроек в app_settings.settings_json — фиксированная схема или произвольный JSON?  
→ **Предложение**: Фиксированная Pydantic схема для валидации, но хранение в JSONB для гибкости будущих изменений

**Q4**: Нужно ли логирование действий пользователя (audit log)?  
→ **Предложение**: На начальном этапе нет, добавить если потребуется для отладки или compliance

**Q5**: Backup стратегия для БД?  
→ **Предложение**: Ежедневный pg_dump с ротацией 30 дней, автоматизация через cron

**Q6**: Поддержка импорта/экспорта задач?  
→ **Вопрос для уточнения**: Включить в MVP или отложить на следующую итерацию?
