## Context

В `tasks-view.ts` реализован общий тулбар с фильтрами по проекту (`filterProjectId`) и клиенту (`filterClientId`). Фильтры передаются через объект `filters` в дочерние виды: `ListView`, `DateBoardView`, `StatusBoardView`. Каждый дочерний вид вызывает `api.getTasks(this.props.filters)`, который отправляет параметры как query string на `GET /api/tasks`.

Бэкендный endpoint `GET /api/tasks` уже поддерживает параметр `list_id` — фильтрация работает на уровне SQL-запроса. Изменения требуются только на фронтенде.

## Goals / Non-Goals

**Goals:**
- Добавить выпадающий список (select) фильтрации по Списку задач в тулбар `tasks-view.ts`
- Передавать `list_id` через `filters` в `DateBoardView` и `StatusBoardView`
- Переиспользовать уже загруженные списки (`api.getLists()`) для заполнения dropdown

**Non-Goals:**
- Изменение бэкенд API (уже поддерживает `list_id`)
- Изменение вида «Списки» (ListView) — там задачи уже сгруппированы по спискам
- Добавление мульти-выбора (несколько списков одновременно)

## Decisions

### 1. Где загружать данные списков

**Решение**: Загружать `api.getLists()` в `TasksView.onMount()` наравне с проектами и клиентами.

**Обоснование**: Тулбар рендерится в `TasksView`, поэтому данные списков нужны именно там. Оба дочерних вида (DateBoardView, StatusBoardView) уже загружают списки самостоятельно для отображения цветовых индикаторов — это не конфликтует, так как запросы кэшируются на уровне браузера.

### 2. Где хранить состояние фильтра

**Решение**: Добавить `filterListId: number | null` в `TasksViewState`, аналогично `filterProjectId`.

**Обоснование**: Единый паттерн с существующими фильтрами. При смене вида фильтр сохраняется.

### 3. Передача фильтра в дочерние виды

**Решение**: Добавить `list_id` в объект `filters` внутри `renderSubView()`.

**Обоснование**: Минимальное изменение — типы `DateBoardViewProps.filters` и `StatusBoardViewProps.filters` расширятся опциональным полем `list_id`. API-клиент уже прокидывает все ключи `filters` как query-параметры.

## Risks / Trade-offs

- **[Дублирование загрузки списков]** → `TasksView` и дочерние виды оба вызывают `api.getLists()`. Это приемлемо: запросы лёгкие, кэшируются браузером. Выносить в общий store — over-engineering для данной задачи.
- **[Фильтр не применяется к ListView]** → В виде «Списки» фильтр по списку избыточен (задачи уже сгруппированы по спискам). Если пользователь переключится на «Списки» с активным фильтром, `list_id` будет передан, но визуально это не вызовет проблем.
